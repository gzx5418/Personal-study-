## 二、V8 新发现问题清单

### P1 级别 - 功能缺陷

| 编号 | 问题 | 详情 | 影响 |
|------|------|------|------|
| P1-15 | **会话消息重复存储** | `chat_agent.py:84-85` 将 assistant 响应写入 session，`chat.py:92-93` 在 SSE 结束后也写入 session。每次对话响应被存两遍，导致历史消息翻倍。 | 对话历史膨胀、上下文浪费 Token |

## 九、V8 新发现 P1 问题详细分析

### P1-15：会话消息重复存储

**现象**：每次对话，assistant 响应在数据库中被存两条。

**根因**：双重写入路径：
- `chat_agent.py:84-85`：Agent 处理完后直接写入 session
- `chat.py:92-93`：SSE 事件流结束后，收集的完整响应再次写入 session

**修复**：删除 `chat_agent.py` 中的 `session_service.add_message()` 调用（第 84-85 行），仅保留 `chat.py` 中 SSE 收集后的写入。`chat.py:60` 已负责写入 user message，Agent 不需要再写。

### P1-16：图片服务路径遍历

**现象**：`GET /api/chat/image/{filename}` 可通过构造 `../../etc/passwd` 等路径读取服务器文件。

**根因**：`chat.py:193` 使用 `os.path.join(img_dir, filename)` 直接拼接 URL 参数。

**修复**：在 `serve_image()` 开头添加校验：
```python
if "/" in filename or "\\" in filename or ".." in filename:
    return {"error": "Invalid filename"}
```

### P1-17：会话列表未按用户过滤

**现象**：`GET /api/chat/sessions/{user_id}` 返回所有用户的所有会话。

**根因**：`session_service.list_sessions()` 的 SQL 查询未按用户过滤。`chat.py:152` 虽接受 `user_id` 参数但未使用。

**修复**：在 SQL 查询中增加用户过滤，或根据 session_id 前缀匹配。
